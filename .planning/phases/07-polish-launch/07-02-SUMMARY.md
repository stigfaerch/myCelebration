## Bugs Found + Fixed During QA

VerifyQA's first test run surfaced **two genuine bugs in my own initial spec
code** (not app bugs — test selector bugs). Investigating and fixing them
before shipping is exactly the root-cause-first methodology this spec
demands.

### Bug 1: `_smoke.spec.ts` expected 404 but received 500

**Symptom**: `expect(response?.status()).toBe(404)` — received 500.

**Root cause**: Path `/nonexistent-route-for-smoke-test` is a single URL
segment. Next.js's `src/app/[uuid]/page.tsx` is a dynamic catch-all for any
single-segment path, so the route *matched*, entered the guest layout,
attempted to `resolveGuest()` + fetch guest-scoped data against the
placeholder Supabase URL (`your-project.supabase.co`), crashed, and was
caught by `src/app/[uuid]/error.tsx` — which returns 500, not 404.

**Fix**: Use a multi-segment path `/__smoke__/does/not/exist` — Next.js has
no dynamic `[...uuid]` catch-all, so multi-segment misses fall through to
the root `not-found.tsx` which correctly returns 404.

**Regression protection**: Fix is in the test itself; subsequent runs catch
the 500-vs-404 mismatch immediately.

### Bug 2: `auth.spec.ts` admin login — `getByLabel` timeout

**Symptom**: `page.getByLabel(/kodeord/i).fill(...)` timed out.

**Root cause**: Inspected `src/app/admin/login/page.tsx` (and
`src/app/[uuid]/enter/page.tsx`). Both forms use `<input placeholder="Kodeord" />`
with **no `<label>` element**. Playwright's `getByLabel` only matches
accessible-name via `<label for>` / `aria-label` / `aria-labelledby`. A
placeholder is not an accessible name.

**Fix**: Switched to the resilient chain
`getByPlaceholder(/kodeord/i).or(getByRole('textbox')).or(input[type="password"])`
for admin (which can have error banners hiding placeholder visibility) and
to `input[type="password"]` directly for guest `/enter` (single input, no
ambiguity).

**Regression protection**: If someone later adds a proper `<label>` the
tests still work (the `.or()` chain + type selector keeps matching). If the
input type is changed away from `password`, that IS a UX regression worth
failing on — test failure would correctly flag it.

**Observation (not an app bug, but accessibility nit)**: Both password
forms have no programmatic label. A real `<label>` (or at least
`aria-label="Kodeord"`) would be a small a11y win. Logged as an
observation, not a launch blocker — password-field semantics + visible
placeholder are sufficient for screen readers in all mainstream browsers,
and the form only has one field so context is unambiguous.

## Decisions

1. **Env-var-gated DB tests (not mocked Supabase).** The plan offered two
   options (mock via MSW vs. gate via env var). I chose env-var gating: (a)
   an MSW layer is ~300 lines of wiring for a Phase-7 polish deliverable,
   (b) mocked tests can green-light broken realtime wiring that only shows
   up against the real service, (c) the QA-MATRIX already requires manual
   two-session testing against the real service for R17 + R23 — the
   automated smoke harness is a trust-but-verify supplement, not a
   replacement.

2. **Chromium-only for v1.** The plan allowed ff/webkit as optional. A
   single-browser matrix minimizes install friction (chromium is ~112MB;
   adding webkit adds another ~150MB, plus Safari-specific flakiness), and
   the guest audience is mixed-device but the critical flows are covered
   by the responsive-viewport matrix already documented in 07-01.
   Post-launch, adding `{ name: 'webkit', use: devices['iPhone 14'] }`
   would be a 2-line addition to `playwright.config.ts`.

3. **Manual-test rows for two-session flows (R17, R23).** Playwright
   supports multi-context tests (two `browser.newContext()` instances) but
   wiring a deterministic race for "guest A creates swap → guest B sees
   it in <1s" requires Supabase realtime against a live project, careful
   ordering of login+subscribe, and assertion timing that is easy to make
   flaky. For Phase 7 this is better as a manual row with explicit
   reproduction steps. Candidate for a post-launch test-hardening pass.

4. **Harness test uses multi-segment 404, not single-segment 404.** See
   Bug 1 above. The one-line lesson: anywhere Next.js has a broad dynamic
   segment like `[uuid]`, smoke tests picking "random" paths have to
   intentionally dodge it.

5. **Placeholder-based password-field selection, not label-based.** See Bug
   2. The app's login forms are label-less; tests have to match reality.

6. **`test:smoke:ui` script included.** One-off debug-mode invocation
   (`npm run test:smoke:ui`) for when a test fails and the operator wants
   the Playwright UI's time-travel inspector. Zero runtime cost when
   unused.
