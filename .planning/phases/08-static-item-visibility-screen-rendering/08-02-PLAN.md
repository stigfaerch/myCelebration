<task id="1" title="Pick a storage layout (Decision B) and write migration 011">
<action>
1. Choose between Option B1 (polymorphic single table — extend `screen_page_assignments`) and Option B2 (separate `screen_static_assignments` table). Document the choice in SUMMARY.md with the trade-off you weighed.

   Recommended bias: **B1**. Reasons:
   - Cycler refactor (Plan 08-05) can fetch one ordered list of mixed items via a single query (clean cycle order across page/static)
   - Single sort_order space avoids merge-sort complexity in the renderer
   - Realtime: the screen subscribes to ONE table topic instead of two
   - Trade-off: schema feels less normalized (page_id and static_key are mutually exclusive), needs a check constraint

   B2 trade-offs: cleaner per-table schema; cycler needs to fetch both tables and merge by sort_order (which is independent across tables, so admins can't naturally interleave); two realtime subscriptions; more redundancy in the action layer.

2. Create `supabase/migrations/011_polymorphic_screen_assignments.sql`. If you chose B1:

```sql
-- myCelebration — Polymorphic screen assignments
-- Apply manually via: Supabase Dashboard > SQL Editor.
--
-- Extends screen_page_assignments to also reference static menu items
-- via static_key. Exactly one of (page_id, static_key) must be non-null.

alter table screen_page_assignments
  add column if not exists kind text not null default 'page',
  add column if not exists static_key text;

alter table screen_page_assignments
  drop constraint if exists spa_kind_check;
alter table screen_page_assignments
  add constraint spa_kind_check
  check (kind in ('page', 'static'));

-- Make page_id nullable (was NOT NULL in migration 008)
alter table screen_page_assignments
  alter column page_id drop not null;

-- Ensure exactly one of (page_id, static_key) is set
alter table screen_page_assignments
  drop constraint if exists spa_target_xor;
alter table screen_page_assignments
  add constraint spa_target_xor
  check (
    (kind = 'page' and page_id is not null and static_key is null) or
    (kind = 'static' and static_key is not null and page_id is null)
  );

-- Replace old unique(screen_guest_id, page_id) with two partial uniques
-- so a screen can't have the same page twice OR the same static-key twice
alter table screen_page_assignments
  drop constraint if exists screen_page_assignments_screen_guest_id_page_id_key;

create unique index if not exists spa_unique_screen_page
  on screen_page_assignments(screen_guest_id, page_id)
  where kind = 'page';

create unique index if not exists spa_unique_screen_static
  on screen_page_assignments(screen_guest_id, static_key)
  where kind = 'static';

-- REPLICA IDENTITY FULL was set in migration 009; remains in effect.
-- Realtime publication membership unchanged (table is already published).
```

   If you chose B2: create new table `screen_static_assignments(id uuid pk, screen_guest_id uuid, static_key text, sort_order int, created_at timestamptz, unique(screen_guest_id, static_key))`. Set RLS, add to `supabase_realtime` publication, set `replica identity full`.
</action>
