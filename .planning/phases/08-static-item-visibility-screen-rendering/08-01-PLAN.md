<objective>
Create the persistence layer and server actions for per-static-item visibility metadata (is_active + visible_from + visible_until). The same fields that live on dynamic `pages` rows must now exist for each static menu key (`tasks`, `camera`, `photos`, `program`, `hvor`, `deltagere`, `minder`, `galleri`). Wave 2 plans (08-03 admin UI, 08-04 guest filter, 08-05 screen render) consume these actions.
</objective>

<context>
- Read 08-CONTEXT.md for full phase context, including the Decision A trade-off (dedicated table vs jsonb column)
- Existing dynamic-page visibility pattern lives on the `pages` table: `is_active boolean default false, visible_from timestamptz, visible_until timestamptz` (see migration 001). Mirror these column names exactly so reusable predicates (`isPageVisibleNow` in `src/lib/guest/navItems.ts`) can be applied unchanged
- The static-key universe is the const `STATIC_NAV_KEYS` in `src/lib/guest/navItems.ts` — currently 8 keys: `tasks, camera, photos, program, hvor, deltagere, minder, galleri`. The schema must NOT hard-code these keys (admin shouldn't have to migrate when STATIC_NAV_KEYS changes); use a free `text` column for `static_key`
- Default behavior: a static item with NO row in the visibility table (or NO entry in the jsonb) is treated as fully visible. This preserves current behavior — no migration backfill needed
- Migrations are applied manually via Supabase Dashboard SQL Editor. Use `create table if not exists` / `add column if not exists` for idempotency. Header comment must instruct manual application
- `getStaticItemVisibilityMap()` is read-only and called from BOTH `getResolvedNavForGuest` (08-04) AND admin UI (08-03). Performance: this is one row per static key (~8), so a single SELECT * is fine — no need for an indexed WHERE
</context>

<action>

   Recommended bias: A1. Reasons:
   - Same column shape as `pages` → reuse `isPageVisibleNow` predicate
   - Easier to add columns later (e.g., per-screen visibility overrides if v2 wants that)
   - Indexable, queryable; jsonb fields require `->>` casts everywhere

2. Create `supabase/migrations/010_static_item_visibility.sql`. If you chose A1:

```sql
-- myCelebration — Static item visibility settings
-- Apply manually via: Supabase Dashboard > SQL Editor.
--
-- One row per static menu key. Absence of a row means "fully visible"
-- (default behavior preserves current state without a backfill).

create table if not exists static_item_settings (
  static_key text primary key,
  is_active boolean not null default true,
  visible_from timestamptz,
  visible_until timestamptz,
  updated_at timestamptz not null default now()
);

-- RLS: service role bypasses; admin server actions use service-role client.
-- No anon access needed (visibility metadata is consumed server-side via
-- `getResolvedNavForGuest` and friends).
alter table static_item_settings enable row level security;
-- Intentionally no policies → anon DENY by default.
```

   If you chose A2: add a `static_visibility jsonb not null default '{}'` column to `app_settings`. Document the shape: `{ "tasks": { is_active: boolean, visible_from: string|null, visible_until: string|null }, ... }`.

<task id="2" title="Create server actions in src/lib/actions/staticItemVisibility.ts">
<action>
1. Create `src/lib/actions/staticItemVisibility.ts` with `'use server'` directive at the top.

2. Define types matching the chosen storage layout:

```ts
export interface StaticItemVisibility {
  static_key: string
  is_active: boolean
  visible_from: string | null
  visible_until: string | null
}
```

3. Implement these exports:

   - `getStaticItemVisibilityMap(): Promise<Record<string, StaticItemVisibility>>` — read-only, no admin assertion (called from guest paths). Returns map keyed by static_key. Static keys NOT in the map are treated as fully visible by callers.

   - `updateStaticItemVisibility(staticKey: string, settings: { is_active: boolean; visible_from: string | null; visible_until: string | null }): Promise<void>` — `await assertAdmin()`. Upsert pattern: if no row exists, create one; if exists, update. For Option A1 use `.upsert({ static_key, ...settings, updated_at: new Date().toISOString() }, { onConflict: 'static_key' })`. For Option A2, read-modify-write the jsonb. Validate that `staticKey` is in `STATIC_NAV_KEYS` (import from `@/lib/guest/navItems`) — reject unknown keys with a clear error.

   - `isStaticItemVisibleNow(staticKey: string, map?: Record<string, StaticItemVisibility>): boolean` — pure helper, NOT async. Looks up visibility metadata in the provided map (or fetches if map omitted). If no entry: return true. Otherwise reuse `isPageVisibleNow` from `@/lib/guest/navItems` with shape adaptation.

   - `revalidatePath` calls: after `updateStaticItemVisibility`, call `revalidatePath('/admin/sider')` AND `revalidatePath('/[uuid]', 'layout')` so guest layouts refetch the nav.

4. Add JSDoc comments to each export explaining the intended consumer (admin UI vs guest path).
</action>
